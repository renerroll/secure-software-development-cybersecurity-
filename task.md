Архітектурне ревʼю шаблону на можливість проведення перевірок і наявність сигналів перевірки



Покрокова інструкція для виконання роботи



Підготовка середовища

1. Створіть локальну папку secure-arch-lab/.

2. Розмістіть у цій папці отримані файли із архіву: 

main.py, deployment.yaml, main.tf, ci.yml, Dockerfile, lab.md, README.md.



Частина 1. Початковий аудит архітектури

1. Відкрийте файл lab.md, ознайомтеся із завданням, форматом звіту та чеклістом.

2. Проаналізуйте кожен із файлів проєкту:

main.py — чи є типізація? логування? OpenAPI?
deployment.yaml — як передаються секрети? чи використовується CSI / Vault?
ci.yml — чи є перевірки? наприклад, SBOM, semgrep, gitleaks?
main.tf — чи обмежене джерело трафіку?
Dockerfile — чи є підпис? SBOM?


Частина 2. Складіть звіт про виявлені проблеми

1. Створіть новий файл у цій папці — report.md.

2. Для кожної з щонайменше 5 виявлених проблем:

Опишіть її у форматі, наведеному в lab.md (вкажіть файл, суть проблеми, порушений принцип, посилання на конспект).
Запропонуйте варіанти виправлення: наприклад, архітектурний патерн, політику, інструмент або IaC-рішення.

Частина 3. Додатково (за бажанням)

Створіть окрему папку fix/ і покладіть туди файл(и) з виправленим кодом.
Або додайте приклад, де покажете різницю: оригінальний файл ➝ виправлений.


Частина 4. Передача результату

1. Перевірте, що у вашій папці є:

report.md — ваш звіт;
(опційно) fix/ — виправлені файли;
уся структура проєкту з файлів лабораторної роботи.


Памʼятайте:

Ваша мета — не просто «пофіксити баги», а показати, що архітектура не дозволяє перевірку.
Архітектурне ревʼю — це аналітичне мислення й технічне обґрунтування.
Використовуйте поняття з конспекту: PoLP, сигнал, патерн, гейт, шаблон, контроль.


Готові? Проведіть самоперевірку по критеріям перед тим, як відправити ДЗ в ЛМС:

Чи виявили ви порушення контрольованості?
Чи показали ви, чому без сигналу система «мовчить»?
Чи ґрунтується ваше виправлення на патерні з конспекту?
Якщо відповіді «так» — ви на правильному шляху!



Результати виконання можна прислати в репозиторії або у Google-документі.
Для коду в Google Docs:

Вставляйте блоки коду як python / yaml або шрифтом Courier New з сірим тлом.
Для додаткового завдання: додайте блок “Виправлений приклад” або лінк на папку fix/.


Формат здачі

Створіть Google-документ / публічний репозиторій із виконаним завданням.
Додайте посилання на документ / репозиторій у LMS.
Завантажте документ / репозиторій із виконаним завданням на свій комп’ютер та прикріпіть його в LMS у форматі zip. Назва: ДЗ[номер]_ПІБ.
